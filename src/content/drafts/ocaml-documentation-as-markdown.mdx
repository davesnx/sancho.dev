`dune 3.22` comes with the `@doc-markdown` alias which generates markdown for your documentation, thanks to odoc.

```bash
dune build @doc-markdown
```

This feature (both the odoc markdown backend and the dune integration) has been implemented by me, which isn't a big flex, and more a testament on what I wanted to see OCaml tooling to be.

Let me explain

## Use case 1: Melange documentation

This was my main motivation.

At Melange, API docs come from odoc, while the rest of the docs site is VitePress. That split causes real problems:

- linking between guides and API pages is harder than it should be
- docs search does not naturally include generated API HTML
- users jump from a unified docs UX to a separate API UX
- header/sidebar/theme/tracking setup gets duplicated

With markdown output, API docs become part of the same pipeline as the rest of the site.

That is the key reason I pushed for this work.

## Use case 2: `mdx` + odoc (executable docs)

Another good use case is running snippets as part of CI.

If your docs source is `.mld`/`.mli` and you run `mdx`, examples stay executable instead of becoming stale text.

Typical stanza:

```lisp
(mdx
 (files README.mld lib/your_lib.mli)
 (libraries your_lib))
```

This gives you a clean setup:

1. odoc remains the source for API structure
2. `mdx` verifies examples

So docs are both generated and tested.

## Use case 3: Parseff README flow

In [parseff](https://github.com/davesnx/parseff), I use `README.mld` as the source and generate `README.md` for GitHub.

The relevant dune setup looks like this:

```lisp
(documentation
 (package parseff)
 (mld_files README))

(mdx
 (files README.mld lib/parseff.mli)
 (libraries parseff))

(rule
 (alias doc-markdown)
 (mode (promote (until-clean)))
 (deps %{workspace_root}/_doc/_markdown/parseff/README.md)
 (targets README.md)
 (action
  (system
   "printf '<!-- Please do not edit this file directly. Update README.mld instead. -->\\n\\n' > %{targets} && tail -n +2 %{workspace_root}/_doc/_markdown/parseff/README.md >> %{targets}")))
```

The flow is straightforward:

- write `README.mld`
- run `dune build @doc-markdown`
- promote the generated `README.md`
- commit generated output

One source, multiple targets, less manual syncing.

## What this enables

- API docs can be integrated into markdown-first static site generators
- publishing to GitHub/GitLab gets simpler
- search can index guides and API docs together
- markdown is easier to feed into retrieval/AI tooling than raw generated HTML

This is mostly an integration improvement, and in practice that is a big win.

## Try it on your project

If you want to test it quickly:

1. Upgrade to dune `3.22`
2. Run `dune build @doc-markdown`
3. Inspect `_doc/_markdown`
4. Add `mdx` where you have runnable examples
5. Optionally add promotion rules for `README.md` or other published files

If you hit edge cases, please report them. Real-world feedback is what will improve these workflows.
